Documentation is like a guide or an aid that can help others use different things depending on what the documentation is meant for. There are many different types of documentation such as Technical, User, and Marketing.

 

Technical documentation is defined as documentation that “describes how a product or service operates.”  A good example of this is software code documentation where the person who writes the code tells the user what each part of the code does in the program.

 

User documentation is documentation that is meant to assist the user with whatever they may be using. It is not documented source code like technical documentation, though. This documentation is usually given with the product like a manual or a user guide.



Marketing documentation is used to inform the user about what they’re using while “exciting” them as well. It describes to them what exactly they will be receiving so that they are not expecting something that the program will not give them.

Documentation is electronic or printed information that can exists for many reasons. In the case of software documentation it can be boiled down to about five different forms. Requirements, Architecture/Design, Technical, End User, and Marketing. 

Requirements is a type of documentation that identifies attributes capabilities, characteristics, or qualities of a system.

Architecture/Design Documentation would be an overview of the software the documentation is for. 

Technical Documentation is the documentation of code, algorithms, interfaces, and APIs.

End User Documentation acts as a manual for the user of the software. It describes how to correctly use the software.

Marketing Documentation would describe how to properly market the product and usually include analysis of the market demand.

In general software documentation describes a piece of computer software. Specific types of documentation can appear when documenting software. Some of the various documentation techniques used in developing software include user documentation, requirements documentation, architecture documentation, and technical documentation.

User documentation is created for the user of the product and describes the proper way to use the software. It is basically the manual for the software and can be in electronic or printed form.

Requirements documentation describes the purpose of the product, what it can do and be used for. Software may need extensive requirements documentation or very little depending on how and what it will be used for.

Architecture documentation is used to record information about the architecture of the software. This documentation can be helpful for anyone involved in the development of the product itself by providing architectural models.

Technical documentation is exactly how it sounds. It describes the technical parts of the software like code, algorithms, API’s, and interfaces.

Documentation is defined as “a set of documents provided on paper, or online, or on digital or analog media such as audio tape or CDs.”  Some examples of documentation are user guides and manuals, online help and quick reference guides distributed through software products, online applications and through websites.  Some types of documentation are Technical documentation, user documentation, legal documentation, and clinical documentation.

Technical documentation describes how a product or service operates. Examples include API documentation, technical specifications and software code documentation.

User documentation is documentation provided to end users to assist in using a product or service. Examples include instruction manuals and online help.

Legal documentation usually comes from courts or government agencies and defines legal rights, duties and privileges of people. Examples include contracts and written agreements.

Clinical documentation is the creation of an analog or digital record detailing a medical treatment, medical trial or clinical test.  MRI’s, X-rays and EKG’s can also accompany this type of documentation.

There are many different types of documentation. The Wikipedia page about software documentation describes 5 different types of documentation. These types are requirements, architecture/design, technical, end user, and marketing.

I have noted different types of documentation. 

The first type of documentation is Requirements documentation. This type of documentation is specific to what everyone wants the software to do. It is an agreed upon format that everyone is included in. Requirement documentation can be more strict depending on what the software is intended for.

Another type of documentation is Technical documentation.  This documentation is just as it stats. It is technical writing about the software that specifies what the software must to as accurately as possible. 

The next type of documentation is User documentation. This type simply describes how the software is used. Such as a tutorial or demo. It is used to help the user get an understanding of the software. 

The fourth type of documentation I would like to talk about is Marketing documentation. This one doest exactly what it is called. It's used to market the software. It shows what it does, and it excites the possible customers and users.

Based on my findings, I could see how many people categorized the types of documentation similarly. I used to think that documentation was simply a piece of paper that related to whatever it was deemed related to, whatever was written on it. After reading through many sources that categorized the documentation types, I can see how there is so much more to the types of documentation. They are more than a piece of paper, they are a file on a computer, a report that was emailed and so much more. While I did not want to use wikiversity.org, I still wanted to include their findings with my other sources I found. Wikiversity.org had categorized the documents into two categories, Process Documents: they guide the development, testing, maintenance and improvement of systems. They are used by managers, engineers, testers, and marketing professionals. These documents use technical terms and industry specific jargon. They include, test plans, design specifications and project plans. The other category is User Documents give customers the information they need to use the product. They are primarily teaching materials which include some technical explanation. They use everyday terms in place of technical jargon, making it easier for the novice or outsider to understand the system. They include descriptive document, installation, configuration, user manual and systems reference.

My second source had a more descriptive list of documentation. Rhyous.com stated that there are eight different types of documentation.

  1. Guides, Walk-throughs
  2. Product feature documentation
  3. Troubleshooting documentation
  4. Knowledge based (problem, cause, resolution)
  5. Code, API or SKD documentation
  6. Internal Development Documentation
  7. Real life customer implementations
  8. Marketing documentation

    If a deliverable is being produced by a small team and said deliverable is intended to be used by a limited number technical users for a short span of time, documentation may not be hugely important.


    However, minute increases in the scope of a project can quickly result in complexity that introduces large project management challenges. At such a juncture, teams may come to regret the technical debt that they incurred at the outset of such projects.


    Documentation may serve various purposes; It can be meant to preserve data accurately, such as in the realm of IT Configuration Management, or it can be intended to communicate the ways the deliverable was meant to be deployed and its limitations, or the material can be meant to be used for marketing purposes.


    Configuration Management documentation can provide us with the data we need to track deployed resources in an IT environment so that we can better extrapolate about future needs and better maintain our current infrastructure.


    Usage documentation can minimize the need for future technical support of a deliverable so that we can focus more on the product's maintenance, and users can solve their own problems without navigating what are usually complex channels of technical support options.


    Marketing documentation can explain to non-technical users that things that a deliverable can do for them, or the business needs that could be met by integrating a deliverable into their workflow.


    Project management, technical suport, and marketing are all necessary evils in the IT industry, but good documentation can beat back some of these challenges.
